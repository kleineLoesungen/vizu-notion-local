# Quick Task 260614-v81: Fix Group-Item Node Deduplication in Mermaid Block Diagrams

**One-liner:** Thread `_groupKey` through group/group-item/nodeId so a component appearing under N parent groups produces N distinct Mermaid node IDs (one per group block), then remove two debug console.log lines.

## Problem

A component with N parent rows (expanded via `expandRelationRows`) appears in N separate `{{#group-item}}` blocks, but `{{title}}` inside always produces the same hash because `nodeId` only scoped by `source`, not by group membership. Mermaid silently placed the node only in the first subgraph block.

## Solution

Three targeted edits to `server/utils/templates.ts`:

1. **`group` helper** — adds `_groupKey: key` to each group object returned from the map, exposing the group's key as a side-channel property alongside the named field.

2. **`group-item` helper** — reads `_groupKey` from the outer `this` context (the group object) and injects it into each item's render context via `options.fn({ ...item, _groupKey: groupKey })`.

3. **`nodeId` helper** — reads `(this as any)?._groupKey` from context, builds scope as `${source}\x00${groupKey}` when `_groupKey` is present. Uses a null-byte separator to prevent hash collisions between combined strings. When `_groupKey` is absent (edge-based templates), scope collapses to `source` — identical to previous behavior.

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Null-byte separator `\x00` between source and groupKey in scope | Prevents hash collisions between concatenated string pairs |
| `_groupKey` defaults to `''` when absent | Backward compatible — nodeId scope collapses to `source` only, matching pre-fix behavior for all edge-based templates |
