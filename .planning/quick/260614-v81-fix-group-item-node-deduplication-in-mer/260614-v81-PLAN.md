<objective>
Fix group-item node deduplication in Mermaid block diagrams by scoping child node IDs to their parent group key.

Problem: A component with N parents (expanded to N rows via expandRelationRows) appears in N separate
`{{#group-item}}` blocks — but `{{title}}` inside `{{#group-item}}` always produces the same hash because
nodeId only scopes by `source`, not by group membership. Mermaid silently places it in only the first block.

Fix: Thread a `_groupKey` value through the group/group-item call chain so nodeId receives a unique scope
per group, making `stableId("Button", "P1") !== stableId("Button", "P2")`.

Also remove two debug `console.log` lines added during development.

Purpose: Correct block diagram rendering when the same item appears under multiple parent groups.
Output: Updated server/utils/templates.ts and server/routes/api/mermaid/[templateId].get.ts
</objective>

  <action>
Three targeted edits to server/utils/templates.ts:

1. **`group` helper** — add `_groupKey: key` to each group object returned from the map:

   Current return:
   ```ts
   return Array.from(map.entries()).map(([key, items]) => ({
     [field]: key,
     items,
   }))
   ```
   New return:
   ```ts
   return Array.from(map.entries()).map(([key, items]) => ({
     [field]: key,
     _groupKey: key,
     items,
   }))
   ```

2. **`group-item` helper** — read `_groupKey` from outer `this` context and inject it into each
   item's render context so nodeId can read it:

   Current type annotation:
   ```ts
   Handlebars.registerHelper('group-item', function(this: { items?: Record<string, string>[] }, options: Handlebars.HelperOptions) {
     const items = this.items ?? []
     return new Handlebars.SafeString(
       items.map((item) => options.fn(item)).join('')
     )
   })
   ```
   New implementation:
   ```ts
   Handlebars.registerHelper('group-item', function(this: { items?: Record<string, string>[]; _groupKey?: string }, options: Handlebars.HelperOptions) {
     const items = this.items ?? []
     const groupKey = this._groupKey ?? ''
     return new Handlebars.SafeString(
       items.map((item) => options.fn({ ...item, _groupKey: groupKey })).join('')
     )
   })
   ```

3. **`nodeId` helper** — read `_groupKey` from `this` context and combine it with `source` to
   form the hash scope. The combined scope string uses a null-byte separator to prevent collisions:

   Current scope line:
   ```ts
   const source = (options?.hash?.source as string) ?? ''
   const id = stableId(String(value ?? ''), source)
   ```
   New scope lines (replace those two lines):
   ```ts
   const source = (options?.hash?.source as string) ?? ''
   const groupKey = (this as any)?._groupKey as string | undefined ?? ''
   const scope = groupKey ? `${source}\x00${groupKey}` : source
   const id = stableId(String(value ?? ''), scope)
   ```

   The `nodeId` helper function signature must also have its `this` binding accessible — confirm it
   is a `function` (not arrow function). It already is in current code:
   `Handlebars.registerHelper('nodeId', function(_attrName, value, options?) { ... })`
   No change needed to the signature itself.

Edge-based templates that do not use `{{#group-item}}` will never have `_groupKey` in context,
so `groupKey` defaults to `''` and scope collapses to `source` — identical to the previous behavior.
