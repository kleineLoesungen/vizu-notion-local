# Milestone v1.1: Mermaid Diagram Templates

**Status:** ✅ SHIPPED 2026-06-13
**Phases:** 5–6
**Total Plans:** 7

## Overview

Admin-defined Mermaid diagram templates that bind to live Notion data from configured sources, rendered client-side. Includes interaction improvements (zoom/pan, related-nodes filter, has-relation filter) and a browser-based MMD editor for template development without container restarts.

## Phases

### Phase 5: Mermaid Diagram Templates

**Goal**: Admin can define Mermaid diagram templates in `config/*.mmd` files that bind to live Notion data from configured sources, rendered client-side via mermaid.js

**Depends on**: Phase 1 (config model + data pipeline), Phase 3 (viz type selector)

**Plans**: 3 plans

Plans:
- [x] 05-01-PLAN.md — Install dependencies (handlebars, gray-matter, mermaid) + server/utils/templates.ts template loader + validate-config.ts extension
- [x] 05-02-PLAN.md — GET /api/mermaid/:templateId route (Handlebars render with live data) + useMermaidTemplate composable + useSourceData extension (hasMermaidTemplates)
- [x] 05-03-PLAN.md — UI integration: Mermaid template buttons in viz type selector + Mermaid render area + error display in [sourceId].vue + human verification checkpoint

**Success Criteria:**
1. Admin creates a `.mmd` file in `config/` with YAML frontmatter declaring `title` and `sources`, restarts container — the diagram title appears as a viz type option in the selector for every source listed
2. Template body uses Handlebars-style bindings (`{{title}}`, `{{#each source-name}}`) substituted with live Notion data at request time
3. Mermaid.js renders the substituted diagram string client-side with no server-side rendering required
4. A template referencing multiple sources combines rows from all listed sources in the template context
5. Invalid templates (bad Mermaid syntax, unknown source names) fail gracefully with a visible error message rather than a blank view

**Goal**: Four targeted improvements to the Phase 5 Mermaid feature — stable node IDs, filter panel full height, related-nodes filter, and D3 zoom/pan

**Depends on**: Phase 5

**Plans**: 4 plans

Plans:
- [x] 06-01-PLAN.md — server/utils/templates.ts: FNV-1a stableId() + nodeId Handlebars helper + body rewrite before compile()
- [x] 06-02-PLAN.md — components/FilterPanel.vue: remove max-height:20rem from list containers, add max-h-screen to outer panel
- [x] 06-03-PLAN.md — Related nodes filter: server _relations field + composable type + FilterPanel show-related button + viz page handler
- [x] 06-04-PLAN.md — D3 zoom/pan on Mermaid SVG: initMermaidZoom() in useMermaidTemplate + 60vh container + zoom hint

**Success Criteria:**

3. A "show related" button per node hides all except that node and its 1-hop Notion-relation neighbours; clicking again resets
4. The Mermaid diagram supports Ctrl+scroll zoom, drag pan, and fit-to-content on load — same interaction model as Metro and Flow

## Milestone Summary

**Key Accomplishments:**
1. Admin-defined `.mmd` template files with YAML frontmatter + Handlebars syntax bind to live Notion data server-side — no code changes needed to add new diagram types
2. Server-side FNV-1a stable node ID generation (`{{attribute}}` → `nXXXXXX["value"]`) eliminates manual ID management for template authors
3. D3 zoom/pan on Mermaid diagrams (Ctrl+scroll, drag) consistent with Metro and Flow interaction model
4. Related-nodes filter (1-hop Notion relations) and "Has relation" filter added across all viz types
5. Filter panel full-height across all viz types
6. Browser-based MMD editor (`/mermaid-editor`) with live server-side data resolution — template authoring without container restarts

**Key Decisions:**
- `{{attribute}}` always outputs full `id["value"]` node definition — no separate ID-only syntax (keeps templates simple)
- Hash input is value-only (not attr+value) — same text always collapses to the same Mermaid node regardless of which field references it
- POST `/api/mermaid/preview` runs full pipeline server-side — editor output is byte-for-byte identical to app rendering
- Relation resolution in templates fetches related page titles via `retrievePage()` with LRU cache
- `_relations` field added to Mermaid API response for client-side 1-hop neighbour computation

**Issues Resolved:**
- Empty `parent` field from relation properties: added async `resolveRelationValues()` to resolve relation roles to page titles
- Duplicate nodes from same text: hash changed to value-only so `{{title}}` and `{{next}}` with same value share one node
- Vue template parse errors with `{{field}}` in editor: resolved with HTML entities `&#123;&#123;field&#125;&#125;`

**Technical Debt:**
- Mermaid viz type not persisted in share links (falls back to metro on arrival) — ViewState.vizType not extended for mermaid
- Template authoring still requires container restart (hot-reload deferred from Phase 5 context)
